# Заняття 7 — Аутентифікація

## Аутентифікація vs авторизація

- Аутентифікація — перевірка хто ти є (логін/пароль)
- Авторизація — перевірка що тобі дозволено робити (права доступу)

## Реєстрація користувачів

- Модель користувача — `email`, `password`, інші поля
- Хешування паролів з `bcrypt` — ніколи не зберігати пароль у відкритому вигляді
- Схема валідації `req.body` через `celebrate`
- Логіка контролера реєстрації — перевірка чи існує користувач, хешування, збереження
- Видалення паролю з відповіді — не повертати `password` клієнту

## Логін користувачів

- Логіка контролера логіну — знайти користувача за `email`
- Порівняння паролів через `bcrypt.compare`
- Повернення помилки `401` якщо пароль невірний

## Сесії

- Навіщо потрібні сесії — зберігання стану аутентифікації
- Модель сесії — `userId`, `accessToken`, `refreshToken`
- Створення сесії при реєстрації та логіні

## Куки

- Налаштування `cookie-parser` middleware
- Захист куків:
  - `HttpOnly` — недоступний для JS у браузері
  - `Secure` — передається лише через HTTPS
  - `SameSite` — захист від CSRF атак
  - `Max-Age` — час життя куки
- Куки які використовуємо: `accessToken`, `refreshToken`, `sessionId`

## Логаут користувачів

- Логіка контролера логауту — видалення сесії з бази даних
- Видалення куків у відповіді

## Послідовність заняття

1. Пояснити різницю між аутентифікацією та авторизацією
2. Створити модель користувача
3. Реалізувати `POST /auth/register` — валідація, хешування паролю, збереження, видалення паролю з відповіді
4. Створити модель сесії
5. Реалізувати `POST /auth/login` — знайти користувача, порівняти паролі, створити сесію
6. Налаштувати `cookie-parser` — пояснити `HttpOnly`, `Secure`, `SameSite`, `Max-Age`
7. Записати `accessToken`, `refreshToken`, `sessionId` у куки
8. Реалізувати `POST /auth/logout` — видалити сесію та куки
9. Протестувати через Postman — реєстрація, логін, логаут
